## 🔍 これは何か

Obsidian RAG Protocol（ORP）は、AIエージェントがObsidianボルトを**永続的な知識層**として利用できるようにするオープンプロトコルです。

ORPは単なるプラグインでも、使い捨てのスクリプトでもありません。ファイルシステムにアクセスできるエージェントなら何でも組み込める**プロトコル仕様**と、それに準拠した**リファレンス実装**のセットです。

Hermes、Claude Code、その他あらゆるエージェントフレームワークが、あなたの個人ノートを読み取り、インデックスし、同期し続けることができます——セッションをまたいでも、エージェントが「忘れる」ことはありません。

**主な特徴:**

- **ゼロ・トークン・オーバーヘッド** — コンテキスト注入時のトークン消費を実質ゼロに
- **インクリメンタル索引** — macOSのiCloud同期によるファイル変更を内容ハッシュで確実に検出
- **双方向プロトコル** — 複数エージェントが同じボルトを共有可能
- **人間が読めて機械も検索できる** — JSONインデックスはツール1回の呼び出しで読める

## 🏗 プロトコル概要

ORPは次の3層アーキテクチャで構成されています：

```
┌─────────────────────────────────────────────────┐
│            AIエージェント（Hermes / Claude Code / 任意）   │
│                                                 │
│  ┌─────────────┐   ┌─────────────────────────┐  │
│  │  ユーザー入力  │ → │ 自動コンテキスト注入（検索）│  │
│  └─────────────┘   └──────────┬──────────────┘  │
│                               │                 │
│                    ┌──────────▼──────────┐      │
│                    │  vault-index.json    │      │
│                    │  （SHA256 インデックス）│      │
│                    └──────────┬──────────┘      │
│                               │                 │
│                    ┌──────────▼──────────┐      │
│                    │   Obsidian ボルト     │      │
│                    │  （.md ファイル群）    │      │
│                    └─────────────────────┘      │
└─────────────────────────────────────────────────┘
```

**データの流れ:**

1. **インデックス構築** — `rebuild-vault-index.py` がボルト内の `.md` ファイルを走査し、SHA256ハッシュを計算して `vault-index.json` を生成
2. **コンテキスト注入** — エージェントが非自明な質問を検出すると、`vault-index.json` を読み込み、キーワードを `aliases` にファジーマッチして該当ノートを自動取得
3. **双方向書き込み** — 各エージェントは専用ディレクトリに書き込み、同じボルト内で知識を共有
4. **自動展開** — wikilinkで参照されたノートを自動で再帰的に読み込み、コンテキスト連鎖を構築

## 🎯 コア設計判断

| 設計判断 | 理由 |
|---|---|
| **SHA256ハッシュ（mtimeではない）** | macOSのiCloud同期はファイルのタイムスタンプを変更してしまう。内容ハッシュなら確実に決定的 |
| **エイリアスは優先度で解決** | Frontmatter → 手動マップ → 前回実行 → フォールバック。自動生成は絶対にしない |
| **パスベースの除外（frontmatterベースではない）** | Frontmatterの除外フラグは書き忘れがち。ディレクトリパターンなら確定的 |
| **デフォルトでインクリメンタル** | 同じハッシュなら抽出をスキップ。CPUを節約し、人間の手編集エイリアスを保護 |
| **Frontmatter駆動の要約** | `summary_points` + `last_action` をYAMLフロントマターから取得。プレーンテキストにフォールバック |
| **JSONインデックス（LLM推論ではない）** | `read_file(vault-index.json)` はツール1回、プロンプトトークンゼロで完了 |

## 🔄 オート・コンテキスト・インジェクション

**v3.4** で導入されたこの機能は、ORPの最も強力な仕組みです。

エージェントは次のように動作します：

1. ユーザーの入力が**非自明な質問**であると判定
2. `vault-index.json` を読み込み
3. ユーザーのキーワードを `aliases` フィールドにファジーマッチ
4. マッチしたノートを自動的にボルトから取得してコンテキストに注入

**結果:** ユーザーが「あの件どうなったっけ？」と聞くだけで、エージェントが自動的に該当ノートを特定して読み込みます。手動でファイルを指定する必要はありません。

## 🤝 双方向コラボレーション

ORPは複数エージェントの共存を前提に設計されています。

**Hermes ⇔ Claude Code の例：**

- 2つのエージェントが1つのボルトを共有
- それぞれが専用のディレクトリ（例: `hermes-knowledge/`、`claude-knowledge/`）に書き込み
- 相手の書き込みをインデックス経由で発見し、参照可能

これは「エージェント同士が同じ知識ベースで協働する」という新しいパラダイムです。各エージェントは自分の作業領域を持ちつつ、相手の出力をシームレスに活用できます。

## 🔗 Skill ⇔ Vault 自動展開

**v3.4** で追加されたこの機能により、スキルとボルトが動的に結びつきます。

エージェントがスキル内でObsidianのwikilink記法（`[[ノート名]]`）に遭遇した場合：

1. リンク先のノートをボルトから自動的に読み込み
2. そのノート内のさらに別のwikilinkも再帰的に展開
3. スキル → ボルト → コンテキストの連鎖が、手動コピーなしで成立

これにより、スキル定義に「関連ノート」を手作業で転記する必要がなくなります。

## 🛠 リファレンス実装

`rebuild-vault-index.py` はORP仕様のリファレンス実装です。

**機能:**

- 指定されたボルトサブディレクトリの `.md` ファイルを走査
- 各ファイルのYAMLフロントマターから `title`, `aliases`, `summary_points`, `last_action`, `status`, `author` を抽出
- SHA256コンテンツハッシュを計算し、前回実行時と比較して変更のあったファイルのみ更新
- `rag_exclude: true` の設定されたファイルを除外
- `vault-index.json` を出力

**使用例:**

```bash
python3 rebuild-vault-index.py \
  --vault ~/Documents/MyVault \
  --output ~/.hermes/vault-index.json \
  --scan wiki/projects wiki/career hermes-knowledge/
```

## 📋 Vault Index スキーマ

`vault-index.json` に含まれる各エントリのフィールド：

- **`_content_hash`** — SHA256コンテンツハッシュ（インクリメンタル更新の基準）
- **`path`** — ボルトルートからの相対パス
- **`title`** — フロントマターのタイトル、またはファイル名から推定
- **`summary_points`** — フロントマターの要約ポイント（リスト）
- **`last_action`** — フロントマターに記録された直近のアクション
- **`status`** — ノートのステータス（例: `active`, `archived`）
- **`author`** — ノートの作成者
- **`aliases`** — ファジーマッチ用のエイリアスリスト
- **`updated`** — 最終更新日時

## 📏 自動インジェクション・ルール

ORPを実装するエージェントは、以下のルールに従う必要があります：

1. **非自明な質問の判定** — ユーザー入力が単なる挨拶や明確な指示でない場合、コンテキスト注入をトリガー
2. **ファジーマッチ** — ユーザーのキーワードを `aliases` フィールドに対してファジーマッチし、スコアの高いノートを優先的に読み込み
3. **インデックス破損時の対応** — `vault-index.json` が破損または欠損している場合、再インデックスを実行してから注入を再試行
4. **インデックス古さの判定** — インデックスのタイムスタンプが閾値（規定: 24時間）を超えている場合、自動的に再インデックスを実行
5. **フォールバック** — マッチするノートが見つからない場合、エージェントは通常の推論にフォールバック（無理に注入しない）
